## 1. 结论摘要（先给结论）

- 一句话目标：为个人产品经理/独立开发者构建一个自动化竞争情报监控代理，用户配置自有产品锚定文档与 5-10 个竞品后，系统日级采集官网与社媒信号，经 html2text + LLM 降噪后与历史快照 diff，有变化时用单次 LLM 解读战略意图并给出对照我方定位的行动建议，有变化则推送飞书+存报告，无变化则熔断退出。
- 本次 In / Out 的边界：In = 监控配置（竞品/信号源/飞书 webhook/自有产品锚定文档）、日级采集（httpx 优先 + Playwright 可选兜底）、html2text + LLM 降噪预处理、快照 diff + 单次 LLM 情报生成（4 字段）、有变化→飞书推送+HTML/MD 报告、无变化→熔断、情报反馈（Negative Few-Shot 注入）；Out = 多租户、团队协作、第三方流量 API、AI 提及度采集、多信号交叉验证、3-Agent 多模型分层（P1）、Slack/邮件通知、实时/小时级监控、价值度分级推送。
- 推荐方案：**"Django 单体 + 日级采集 + html2text/LLM 降噪 + diff 熔断 + 单次 LLM 直出情报 + 有变化推飞书"最小闭环**——用户在 Django Admin 配置竞品与自有产品锚定文档后，django-apscheduler 日级触发 httpx 采集 → html2text 初滤 + LLM 降噪 → 存快照（append-only）→ 与上版 diff → 无变化熔断退出 → 有变化则单次 LLM 调用（注入产品锚定文档）生成 4 字段情报 → 渲染 HTML/MD 报告 → 推送飞书卡片 → 用户可反馈。
- 优先验证点：V-004（LLM prompt 模板与降噪/情报生成准确性）、V-016（产品锚定文档解析与注入）、V-017（产品锚定对建议质量的提升）。

## 2. 推荐方案

- 方案名：**竞争情报最小闭环代理（Competitive Intel Minimal Loop Agent）**
- 主流程 / 关键机制：
  1. **配置**（Django Admin）：用户登记自有产品锚定文档（`self_product_doc` 文本粘贴，Nullable；或上传 .md 文件，存路径）+ 5-10 个竞品 URL（`competitor_urls` 为 JSON 数组，每项 `{"url":"...", "title":"链接说明"}`，title 作为每段网页爬取的数据标题）；配置飞书 webhook；选择调度周期（`cron` 字段，django-apscheduler cron 表达式）。`refined_rules` 字段 P1 占位，MVP 不写入。
  2. **采集**（django-apscheduler 日级触发）：对每个信号源 httpx GET → 解析 HTML → html2text 转换为 Markdown 初步过滤 → **LLM 语义降噪**（过滤导航/页脚/广告等噪音）→ 存为快照（append-only，纯文本+关键 meta）。若 httpx 拿不到有效内容（SPA/JS 渲染），可选降级为 Playwright 兜底。
  3. **变化识别**（diff + 熔断）：取该信号源上一条快照与本次 diff；diff 为空 → **熔断**，写入 `IntelligenceFeed` 表（`job_status=NO_CHANGE`），不推送不生成情报；diff 非空 → 进入情报生成。
  4. **情报生成**（单次 LLM 直出）：将 diff 片段 + 自有产品锚定文档作为 prompt 上下文，一次 LLM 调用输出 4 字段结构化情报——变化摘要 / 战略意图解读 / 行动建议 / 证据 diff 片段。证据 diff **嵌入在 `change_summary` 文本或报告渲染素材中**，不独立为 DB 字段（DB-1 裁决）。
  5. **渲染与分发**：情报通过 Jinja2 模板渲染为 HTML 网页报告 + MD 表格文档（报告落盘，`IntelligenceFeed` 存 `html_report_path`/`md_table_path` 路径索引）；同时推送飞书高级卡片（含变化摘要 + 在线预览/下载按钮）。有变化即推送，无分级。`job_status=CHANGED`。
  6. **消费与反馈**：用户通过独立 HTML 网页或 Django Admin 浏览情报报告（收件箱仅展示 `job_status=CHANGED` 的记录，`NO_CHANGE`/`ERROR_CRAWL` 仅在 Django Admin 调度日志可见）；用户可点"毫无意义"并输入评语 → `user_feedback=-1` + `user_comment` 持久化 → 下次情报生成 LLM 启动前作为 Negative Few-Shot 注入 prompt 底部。
- 关键边界/取舍：
  1. **单用户、Django 单体**：无多租户、无团队权限、无协作（Q1 裁决 + 部署形态定稿）。
  2. **2 类信号源，httpx 优先 + Playwright 可选兜底**：官网 + 社媒；httpx 覆盖静态网站，Playwright 仅对 SPA 按需降级（修订-1 推翻 Q2 原约束）。
  3. **html2text + LLM 降噪预处理**：采集后先 html2text 初滤再 LLM 语义降噪，存干净快照；此 LLM 调用独立于情报生成调用（修订-4 新增）。
  4. **diff + 熔断**：diff 非空才触发情报生成 LLM；diff 为空熔断退出，零推送（Q3 裁决）。
  5. **单次 LLM 直出情报**：情报生成阶段一次调用完成判定+解读+建议，注入产品锚定文档（Q5 裁决维持 + 修订-5）。
  6. **有变化则推送，无分级**：has_change=True → 推飞书+存报告；has_change=False → 熔断。价值度字段已移除（修订-3 推翻 Q6 分级推送）。
  7. **全量快照 append-only + 报告落盘**：快照存 DB（append-only）；报告存文件系统 + DB 路径索引（Q7 裁决 + 修订-6 分层）。
  8. **自有产品锚定 P0**：每个监控任务关联产品锚定文档，注入情报生成 prompt（修订-5 新增）。
  9. **Negative Few-Shot 反馈注入**：反馈不只存储，直接影响下次 LLM 推理（修订-8 升级）。
  10. **django-apscheduler 调度**：Django 进程内调度，支持 cron 表达式配置，非消息队列（修订-4 升级）。
  11. **competitor_urls JSON 数组**：每项 `{"url":"...", "title":"..."}`，title 作为爬取数据标题，标识内容来源（DB-4 裁决）。
  12. **self_product_doc Nullable**：产品锚定文档字段允许为空（用户可只上传文件），字段名从 `our_product_doc` 改为 `self_product_doc`（DB-2 裁决）。
  13. **NO_CHANGE 进 IntelligenceFeed**：熔断记录与情报记录同表，通过 `job_status` 区分；收件箱仅展示 `CHANGED`（DB-3 裁决）。
  14. **evidence_diff 不独立建列**：证据 diff 嵌入 `change_summary` 或报告渲染素材（DB-1 裁决）。
  15. **refined_rules P1 占位**：字段保留但 MVP 不写入，属后续评分反馈机制（DB-4b 裁决）。
- 为什么选它（可追溯到证据）：
  1. raw.md「快速实现 MVP」「不要求复杂」→ 最小信号集 + 单次 LLM + 日级调度，最短路径验证价值链。
  2. raw.md UR-3「持续自动采集」「自动识别有意义变化」→ 日级采集 + diff 熔断，自动化闭环。
  3. raw.md UR-5「可决策的行动建议」→ LLM 输出含行动建议 + 证据 diff + **产品锚定注入**，建议个性化可追溯。
  4. raw.md UR-6「主动通知」→ 有变化即推飞书卡片，主动到达。
  5. raw.md UR-7「反馈改进」→ Negative Few-Shot 注入，反馈直接驱动 LLM 改进。
  6. raw.md「给出总体架构建议以及技术栈」→ 部署形态与技术栈已定稿（Django 单体 + 完整技术栈）。

## 3. 备选方案

### 3.1 备选方案：规则阈值引擎（无 LLM 判定）

- 核心机制：用文本相似度/DOM 结构变化/关键词增删等规则判定"有意义变化"，不依赖 LLM。
- 主流程：采集 → 规则计算相似度/变化量 → 超阈值则生成情报（模板填充）→ 推送。
- 边界与取舍：无 LLM 成本；但噪音过滤差（广告/时间戳/A-B 测试误报多）；规则维护负担重；与"自然语言解读"衔接生硬。
- 适用前提（何时会选它）：LLM 成本敏感到不可接受；或目标场景信号高度结构化（如纯价格监控）。
- 不选原因：raw.md 明确要求"自然语言解读战略意图"，规则引擎无法产出意图解读与行动建议；噪音过滤维护成本与 MVP"快速"目标冲突（Q3 裁决）。

### 3.2 备选方案：3-Agent 多模型分层架构（P1 优化方向）

- 核心机制：Agent1（GPT-4o-mini）脱水降噪 → Agent2（GPT-4o-mini）语义差异审计+熔断 → Agent3（Claude 3.5）战略推理，三步独立 LLM 调用，便宜模型过滤、强模型推理。
- 主流程：采集 → Agent1 降噪存快照 → Agent2 审计 diff 熔断 → Agent3 推理生成情报 → 推送。
- 边界与取舍：成本分层优化（便宜模型做过滤，强模型只处理有变化的）；但编排复杂度上升，3 次独立 LLM 调用链路需维护状态与错误处理。
- 适用前提（何时会选它）：MVP 已验证单次 LLM 闭环；LLM 调用成本成为瓶颈；需对不同环节用不同模型优化成本。
- 不选原因：MVP 阶段用户明确选择单次 LLM 直出（修订-2 维持 Q5）；3-Agent 列为 P1 优化项，MVP 先验证价值链。

### 3.3 备选方案：云端 SaaS 多租户版

- 核心机制：部署为多租户 SaaS，支持团队协作、多客户隔离、按量计费。
- 主流程：注册租户 → 配置竞品 → 云端调度采集 → 情报按租户隔离 → 团队成员协作消费。
- 边界与取舍：可商业化、可扩展；但需多租户隔离、认证、权限、计费、高可用；MVP 范围扩大数倍。
- 适用前提（何时会选它）：目标用户为团队/机构（Q1 选项 B/C）；需商业化收费。
- 不选原因：Q1 裁决为个人单用户；raw.md「快速实现 MVP」；多租户属 P2 商业化阶段（Q1 裁决）。

  - 风险/假设：用户上传的 .md/.html 产品文档可能格式杂乱，解析后注入 prompt 的效果不确定
  - 方法：实现时支持 .md 上传/粘贴；html2text 解析 .html；解析后存为结构化文本；情报生成时作为 prompt 上下文注入（截断到合理长度，如 2000 token）
  - 成功/失败信号：产品锚定文档可被解析为干净文本；注入后 LLM 输出的行动建议包含"对照我方定位"的个性化内容
  - Owner：FS（解析+注入实现）+ PM（验收标准）
  - 截止：I2 实现阶段
  - 触发动作：若文档过长导致 prompt 超限，实现截断策略（取产品定位/核心功能/定价段落）；若解析失败，提示用户重新上传

- **V-017** 产品锚定对建议质量的提升验证（新增）
  - 风险/假设：注入产品锚定文档后，行动建议质量是否真的提升，还是只是增加了 prompt 长度
  - 方法：A/B 对比测试——同一竞品变化，分别用"有锚定"和"无锚定"生成情报，由用户盲评建议质量
  - 成功/失败信号：有锚定的建议被用户评为"更相关/更可执行"的比例 >60%
  - Owner：PM（设计评估）+ 用户（盲评）
  - 截止：I2 实现阶段首批情报生成后
  - 触发动作：若提升不显著，考虑简化锚定注入或调整为只在 Agent3 推理时注入关键段落

- **V-018** Negative Few-Shot 注入策略（新增）
  - 风险/假设：反馈积累后 prompt 长度膨胀；注入选取策略未定义（全量/最近 N 条/相似度检索）
  - 方法：MVP 默认注入最近 5 条"毫无意义"反馈作为 Negative Few-Shot；超过 5 条时取最近 5 条
  - 成功/失败信号：注入后 LLM 不再产出类似被标记"毫无意义"的情报；prompt 不超模型上下文窗口
  - Owner：FS（注入实现）+ PM（验收标准）
  - 截止：I2 实现阶段
  - 触发动作：若 prompt 超限，减少注入条数；若"毫无意义"反馈仍频繁出现，升级为相似度检索 top-K 注入

### 7.2 需遵守的不变量

- 快照存储必须 append-only，不得覆盖或轮替历史版本（Q7 裁决）。
- 快照存储内容为 LLM 降噪后的干净 Markdown 文本，不存原始 HTML（修订-4 + V-011）。
- 降噪 LLM 调用与情报生成 LLM 调用是独立的两次调用，不得合并（修订-4）。
- 情报生成 LLM 调用仅在 diff 非空时触发，不得全量调用（Q3+Q4 裁决，成本约束）。
- 情报输出必须包含 4 字段（变化摘要/意图/建议/证据 diff），不得缺字段，不得包含价值度字段（Q5 裁决 + 修订-3）。证据 diff 嵌入 `change_summary` 或报告渲染素材，不独立为 DB 字段（DB-1 裁决）。
- 有变化（has_change=True）即推送飞书 + 存报告；无变化（has_change=False）即熔断退出，不推送不生成情报（修订-3）。熔断记录写入 `IntelligenceFeed`（`job_status=NO_CHANGE`），不独立 JobLog 表（DB-3 裁决）。
- 收件箱/报告浏览仅展示 `job_status=CHANGED` 记录；`NO_CHANGE`/`ERROR_CRAWL` 仅在 Django Admin 调度日志可见（DB-3 裁决）。
- 采集方式 httpx 优先，Playwright 仅对 SPA 按需降级，不得默认全量 Playwright（修订-1）。
- 调度限 django-apscheduler 日级，不得引入消息队列或实时架构（Q4 裁决 + 修订-4）。
- 每个监控任务必须关联一个产品锚定文档（`self_product_doc`，Nullable 允许只上传文件），情报生成时必须注入该文档作为 prompt 上下文（修订-5 + DB-2 裁决）。
- `competitor_urls` 必须为 JSON 数组，每项 `{"url":"...", "title":"..."}`，title 作为爬取数据标题标识来源（DB-4 裁决）。
- Negative Few-Shot 注入上限最近 5 条，超过则取最近 5 条，不得全量注入导致 prompt 超限（V-018）。
- `refined_rules` 字段 P1 占位，MVP 不写入（DB-4b 裁决）。

### 7.3 跨模块影响

- 配置管理 → 采集器：采集器从配置获取竞品/信号源 URL 列表。
- 配置管理 → 产品锚定：产品锚定文档从配置获取，关联到监控任务。
- 采集器 → 降噪引擎：采集器输出原始 HTML，降噪引擎输入。
- 降噪引擎 → 快照存储：降噪后干净文本写入快照表（append-only）。
- 快照存储 → 变化识别引擎：diff 引擎依赖快照表的"最新一条"作为对比对象。
- 变化识别 → 情报生成：diff 非空才触发 LLM；diff 片段作为情报的证据字段。
- 产品锚定 → 情报生成：产品锚定文档作为 prompt 上下文注入，使行动建议个性化。

- 报告渲染 → 文件系统/DB：报告文件落盘，DB 存路径索引。
- 反馈系统 → 情报生成：Negative Few-Shot 在下次情报生成 LLM 启动前注入 prompt。
