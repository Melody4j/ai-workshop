# 自动化竞争情报监控代理

## 需求描述

构建一个自动化的竞争情报监控代理，用户只需设定监控目标，系统即可：

- 持续采集竞品的多维度信号（网站、社媒、流量、AI 提及度等）。
- 自动识别有意义的 变化，并生成结构化情报。
- 用自然语言解读变化背后的战略意图，并给出行动建议。
- 主动、及时地将情报推送给用户。

## 用户需求

| 编号 | 需求 | 描述 |
| ---- | ---- | ---- |
| UR-1 | 快速建立监控 | 用户能够方便地登记自有产品与关注的竞品，并尽快开始监控 |
| UR-2 | 自动发现监控目标 | 系统能根据用户产品自动推荐潜在竞品与相关监控范围，降低配置成本 |
| UR-3 | 持续、自动地采集与解读 | 系统周期性采集竞品信号，自动识别变化并解读其含义，无需人工介入 |
| UR-4 | 结构化情报消费 | 用户能在统一收件箱中浏览、筛选、归档情报，并查看完整分析 |
| UR-5 | 可决策的行动建议 | 情报不仅说明"发生了什么"，还应说明"我该怎么做"及原因 |
| UR-6 | 主动通知 | 用户无需频繁登录，系统可通过邮件 / Slack 等渠道主动推送重要情报 |
| UR-7 | 反馈与改进 | 用户可对情报质量反馈，帮助系统持续改进 |

## 产品愿景

- 用户完成监控配置后，无需人工操作即可定期收到有价值的情报。
- 情报具备"变化—意图—建议"的完整解读，而非原始数据堆砌。
- 用户反馈机制能驱动情报质量持续提升。

## 约束

- 项目不要求复杂，要求快速实现 MVP 版本。
- 基于这个线路给出总体架构建议以及技术栈。

### R1-Q3：「有意义的变化」判定口径（已裁决）

- 本轮结论（可直接引用到 solution）：
  - 变化识别引擎 = 快照 diff + LLM 判定（两段式）。
  - 第一段：系统保存上次采集的 HTML/文本快照，与本次做 diff；diff 为空则跳过。
  - 第二段：diff 非空时交给 LLM 判断"是否为有意义变化"，过滤广告/时间戳/随机 token/A-B 测试等噪音。
  - diff 结果作为"变化证据"保留，支撑情报可追溯性（UR-5）。
- 本轮新增/更新的约束：
  1. 必须持久化历史快照（至少上一版），否则 diff 无对象 → 影响 Q7 持久化范围决策。
  2. LLM 调用时机：仅 diff 非空时触发，控制成本。
  3. 变化识别与情报解读可复用同一次 LLM 调用（diff 判定 + 意图解读合并），降低延迟与成本。
  4. 不采用纯规则阈值（噪音过滤维护成本高，与自然语言解读衔接生硬）。
- 关键决策：
  - 决策点：变化判定口径 → 选择 A（diff + LLM），非 B（纯规则）/ C（全量 LLM 比对）。
  - 理由：raw.md「自动识别有意义变化」+「自然语言解读」；LLM 本就要用于解读，复用到判定成本低；diff 非空才调用，5-10 竞品规模成本可控；diff 可作证据展示。
- 遗留歧义：
  - diff 的粒度（全文文本 diff / DOM 结构 diff / 关键区域 diff）→ 转验证项 V-003（MVP 默认全文文本 diff，后续可优化为 CSS 选择器定位关键区域）。
  - LLM 判定"有意义"的 prompt 标准与阈值 → 转验证项 V-004（实现时设计 prompt 模板并用真实样本验证）。

### R1-Q4：采集频率与实时性（已裁决）

- 本轮结论（可直接引用到 solution）：
  - 采集频率：日级，每个采集任务每天执行 1 次（低峰期如凌晨）。
  - 调度方式：简单 cron 定时器，不使用消息队列/流式架构。
- 本轮新增/更新的约束：
  1. 调度器 = cron/定时器，无需分布式任务队列（Celery/Redis Queue 等）。
  2. 采集量：10-20 任务/天，单实例可承载；无需水平扩展。
  3. LLM 调用量：每天最多 10-20 次（仅 diff 非空时），成本可控。
  4. 时效性口径：情报时效 = 日级（T+1 内推送即满足 UR-3「周期性采集」语义）。
  5. 不做实时/小时级监控（P1 候选升级项）。
- 关键决策：
  - 决策点：采集频率 → 选择 A（日级 cron），非 B（小时级）/ C（实时）。
  - 理由：raw.md「快速实现 MVP」；竞品官网/社媒变化通常以天/周为单位，日级不漏有意义变化；cron 最简，无需消息队列；成本极低。
- 遗留歧义：
  - cron 具体执行时间点（固定凌晨 / 可配置）→ 转验证项 V-005（MVP 默认固定凌晨，如 03:00，后续可开放配置）。
  - 采集失败的重试策略 → 转验证项 V-006（MVP 默认简单重试 1-2 次，失败记录日志，不阻塞后续任务）。

### R1-Q5：情报解读深度与可信度（已裁决）

- 本轮结论（可直接引用到 solution）：
  - 情报解读方式：单次 LLM 直出（MVP 阶段）。
  - diff 判定 + 意图解读 + 行动建议合并到一次 LLM 调用完成。
  - 输出结构化情报：变化摘要 / 战略意图解读 / 行动建议 / 证据 diff 片段。
  - 用户明确「MVP 先用 A」，多信号交叉验证（B）与引用溯源（C）为 P1 候选。
- 本轮新增/更新的约束：
  1. LLM 编排 = 单次调用，无多步推理/无多信号聚合层。
  2. 情报可追溯性靠"证据 diff 片段"展示，不要求 LLM 输出引用标注。
  3. 情报输出 schema 固定 4 字段：变化摘要、意图解读、行动建议、证据 diff。
  4. 单信号解读闭环（一个竞品的一个信号源变化 → 一条情报）；多信号关联留 P1。
  5. prompt 质量直接决定情报质量，需在实现时用真实样本验证（与 V-004 合并）。
- 关键决策：
  - 决策点：解读深度 → 选择 A（单次直出），非 B（多信号交叉）/ C（引用溯源）。
  - 理由：raw.md「快速实现 MVP」；单次调用最短路径；diff 本身即证据，展示片段满足基本可追溯；B/C 属 P1 高价值升级。
- 遗留歧义：
  - 情报输出 schema 的精确字段定义与格式（JSON / Markdown）→ 转验证项 V-007（实现时定义 schema 并与前端消费对齐）。
  - LLM 模型选择（GPT-4o / Claude / 本地模型）→ 留给 FS 技术选型决策，PM 不越权。

### R1-Q6：通知策略与降噪（已裁决）

- 本轮结论（可直接引用到 solution）：
  - 通知策略：分级推送（A 的简化版）。
  - 价值度分档：二档（高 / 低），由 LLM 在情报输出时一并标注。
  - 价值度高 → 推送飞书 webhook；价值度低 → 不推送，仅进收件箱。
  - MVP 通知渠道：仅飞书 webhook（不支持 Slack / 邮件）。
- 本轮新增/更新的约束：
  1. 通知引擎仅需对接飞书 webhook 一个渠道，无需邮件/Slack 适配层。
  2. LLM 输出 schema 在 Q5 的 4 字段基础上增加「价值度（高/低）」字段，共 5 字段：变化摘要 / 意图解读 / 行动建议 / 证据 diff / 价值度。
  3. 价值度=高 → 触发飞书 webhook 推送；价值度=低 → 仅写入收件箱。
  4. 飞书 webhook URL 由用户配置（单用户场景，单一 webhook 即可）。
  5. Slack / 邮件推送为 P1 候选；三档价值度为 P1 候选。
- 关键决策：
  - 决策点：通知策略 → 选择 A 简化版（飞书 + 二档），非 B（全量推送）/ C（不推送）/ A 原版（三档+多渠道）。
  - 理由：用户明确指定；飞书 webhook 实现最简（单 HTTP POST）；二档降低 LLM 判定复杂度；分级推送兼顾「主动通知」与「不打扰」。
- 遗留歧义：
  - 飞书推送消息的格式（卡片 / 纯文本 / Markdown）→ 转验证项 V-008（实现时与飞书 webhook 协议对齐，MVP 默认 Markdown 纯文本）。
  - 价值度判定的 prompt 标准与边界 → 转验证项 V-009（与 V-004 合并：实现时设计 prompt 模板并用真实样本验证价值度判定准确性）。
  - 飞书 webhook 推送失败的降级策略 → 转验证项 V-010（MVP 默认失败重试 1-2 次，失败记录日志，情报仍进收件箱不丢失）。

### R1-Q7：MVP 数据持久化范围（已裁决）

- 本轮结论（可直接引用到 solution）：
  - 持久化策略：全量快照历史 + 情报记录 + 用户反馈，每次采集都存。
  - 快照表：每次采集都新增一条记录（不轮替、不覆盖），保留所有历史版本。
  - 情报表：每条情报持久化，关联竞品 + 信号源 + 快照对（新旧两版快照 ID）。
  - 反馈表：用户对情报的反馈持久化，关联情报 ID（UR-7）。
- 本轮新增/更新的约束：
  1. 快照存储为只追加（append-only），不做两版轮替；diff 时取"最新一条"与"本次采集"对比。
  2. 5-10 竞品 × 10-20 信号源 × 日级采集 → 每天新增 10-20 条快照，年累积约 3650-7300 条；单库可承载，但需考虑存储清理策略（P1）。
  3. 情报记录必须关联快照对（旧快照 ID + 新快照 ID），保证证据可回溯。
  4. 反馈表字段至少包含：情报 ID、反馈类型（有用/无用/补充）、反馈内容、时间戳。
  5. 存储模型：快照表 + 情报表 + 反馈表（+ 配置表：竞品/信号源/飞书 webhook），单数据库即可。
- 关键决策：
  - 决策点：持久化范围 → 选择 B（全量快照历史），非 A（仅两版轮替）/ C（不存快照）。
  - 理由：用户明确指定保留所有历史；全量历史支持回溯任意版本变化趋势与 LLM 判定审计；存储增长可控（日级采集量小）。
- 遗留歧义：
  - 快照存储格式（原始 HTML / 提取后纯文本 / 两者都存）→ 转验证项 V-011（MVP 默认存提取后纯文本 + 关键 meta，原始 HTML 可选；FS 实现时定）。
  - 存储清理策略（永久保留 / N 天后归档 / 容量阈值清理）→ 转验证项 V-012（MVP 默认永久保留，P1 再定归档策略）。
  - 数据库选型（SQLite / PostgreSQL / 其他）→ 留给 FS 技术选型决策，PM 不越权。

### R1-修订：技术设计文档冲突裁决（5 项）

> 背景：用户提供详细技术设计文档（Django Admin + 独立 HTML 消费页 + 飞书卡片），与 R1 Q1-Q7 部分裁决冲突。经 PM 逐条对比分析后，用户对 5 项冲突逐一裁决如下。

#### 修订-1：Q2 采集方式（推翻原约束）

- 原裁决：仅 HTTP GET + HTML 解析，不使用无头浏览器。
- 新裁决：**httpx 优先，Playwright 作为可选兜底**（仅当遇到 JS 渲染的 SPA 网站时）。
- 理由：大多数静态网站用 httpx 即可；社媒等 SPA 网站需要 Playwright 兜底。MVP 先用 httpx，Playwright 列为可选。
- 影响更新：solution.md Q2 约束、采集器模块不变量、V-001（社媒 JS 渲染兜底已覆盖）。

#### 修订-2：Q5 LLM 编排（维持原裁决）

- 原裁决：单次 LLM 直出（判定+解读+建议合并）。
- 新裁决：**MVP 仍采用单次 LLM 直出**，3-Agent 架构列为 P1 优化项。
- 理由：用户明确 MVP 先验证单次调用闭环；3-Agent 多模型分层为 P1 成本优化方向。
- 影响更新：无（Q5 不变）。

#### 修订-3：Q6 推送策略（推翻分级推送）

- 原裁决：二档价值度（高/低），高→推飞书，低→仅收件箱。
- 新裁决：**取消分级推送，改为有变化则推送**。has_change=True → 推飞书+进收件箱；has_change=False → 熔断退出不推送。**价值度字段从 schema 移除**。
- 理由：用户认为变化本身就是推送信号，无需额外分档；简化 LLM 输出 schema。
- 影响更新：情报输出 schema 从 5 字段回到 4 字段（变化摘要/意图/建议/证据 diff）；通知引擎简化（无分级逻辑）。

#### 修订-4：Agent1 降噪步骤（新增，不冲突）

- 新增内容：采集后先用 html2text 做初步过滤，再由 LLM 做语义降噪，存为快照。此降噪 LLM 调用与 Q5 的情报生成 LLM 调用是**独立的两次调用**。
- 与 Q5 的关系：Q5 仍是"单次 LLM 直出"——指的是情报生成阶段（判定+解读+建议合并为一次）。Agent1 降噪是采集阶段的预处理，不计入 Q5 的"单次"。
- 影响更新：执行链路新增预处理步骤；LLM 调用从每天最多 10-20 次增至 10-20 次（降噪）+ 变化时额外调用（情报生成）。

#### 修订-5：自有产品锚定（新增 P0 功能）

### R1-修订：数据库设计审查裁决（4 项）

> 背景：用户提供 3 张表设计（MonitorProject / DataSnapshot / IntelligenceFeed），PM 从需求语义审查发现 12 项问题。用户对关键项裁决如下。

#### DB-1：evidence_diff 字段（不新增）

- 裁决：**不新增 `evidence_diff` 字段**。证据 diff 嵌入在 `change_summary` 或 HTML/MD 报告中呈现，不独立存储为 DB 字段。
- 影响：solution.md 不变量"4 字段"中的"证据 diff 片段"是 LLM 输出的一部分，嵌入在 `change_summary` 文本中或作为报告渲染素材，不单独建列。

#### DB-2：self_product_doc 字段（重命名+改 Nullable）

- 裁决：`our_product_doc` → 重命名为 **`self_product_doc`**，类型改为 **TEXT Nullable**（允许用户只上传文件不粘贴文本时该字段为空）。
- 影响：MonitorProject 表字段名更新；NOT NULL 约束放宽。

#### DB-3：NO_CHANGE 进 IntelligenceFeed（不独立 JobLog）

- 裁决：**`NO_CHANGE` 记录写入 `IntelligenceFeed` 表**，不独立建 JobLog 表。熔断记录与情报记录同表，通过 `job_status` 区分。
- 影响：收件箱/报告浏览需按 `job_status` 过滤（只展示 `CHANGED`）；`NO_CHANGE`/`ERROR_CRAWL` 记录仅在 Django Admin 调度日志中展示。

#### DB-4：competitor_urls 改 JSON + refined_rules 保留（裁决）

- 裁决-4a：`competitor_urls` 从 TEXT（换行分隔）改为 **JSON 数组**，对象格式：`{"url": "链接", "title": "链接说明，作为每段网页爬取的数据标题"}`。每个 URL 携带 title 标题，可标识来源，解决 A3（快照缺信号源标识）与 C1（竞品与信号源混淆）。

- 影响：competitor_urls 数据结构变更；DataSnapshot 的 raw_markdown/clean_markdown 可按 title 标注每段内容来源；refined_rules 列为 P1 功能占位，MVP 不写入。
